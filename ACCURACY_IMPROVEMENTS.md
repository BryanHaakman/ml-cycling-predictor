## Idea 1: Rank-Gap-Weighted Pair Sampling + Sample Weights

**Category:** Data/Sampling Strategy
**Expected AUC gain:** +0.005 to +0.015 (combined)
**Training time impact:** ~0%

### What

Two complementary changes that cost zero extra training time:

#### Part A: Weighted Pair Sampling (`data/builder.py`)

Replace uniform random pair sampling with **weighted sampling** where each candidate pair's weight is `1 / (|rank_a - rank_b| + 1)`. This shifts the distribution toward close-rank (hard) pairs while keeping `pairs_per_stage=200` unchanged.

```python
# In build_pairs_sampled, replace uniform sampling with:
all_candidates = [(i, j) for i in range(n) for j in range(i+1, n)]
gaps = [abs(riders[i][1] - riders[j][1]) for i, j in all_candidates]
weights = [1.0 / (g + 1) for g in gaps]
sampled = random.choices(all_candidates, weights=weights, k=pairs_per_stage)
```

#### Part B: Sample Weights During Training (`data/builder.py` + `models/benchmark.py`)

Store the rank gap and date with each pair, then compute composite sample weights:

```python
# In builder.py — add rank_gap and stage_date columns
all_pairs.append({
    ...,
    "rank_gap": abs(rank_better - rank_worse),
    "stage_date": stage["date"],
})

# In benchmark.py — compute and pass sample weights
rank_weight = 1.0 / (pairs_df["rank_gap"] + 1)
days_ago = (pd.Timestamp.now() - pd.to_datetime(pairs_df["stage_date"])).dt.days
recency_weight = np.exp(-days_ago / 730)  # half-life ~2 years
sample_weight = rank_weight * recency_weight
sample_weight /= sample_weight.mean()  # normalize

xgb_model.fit(X_train_scaled, y_train, sample_weight=train_weights)
```

### Why This Helps

With uniform top-50 sampling, the **average rank gap is ~17 positions**. A rank-1 vs rank-48 pair is trivially predictable — the model gets it right with >95% confidence and learns nothing from it. Roughly **60% of training capacity** is wasted on these "easy" examples.

In real deployment, you're comparing two competitive riders for the same podium spot. Weighted sampling focuses the training distribution on gaps of 1–10, which are the **hard, informative examples** that define the decision boundary. This is essentially **curriculum learning** — focusing on examples at the model's competence frontier.

The recency component also helps because the test set is 2025–2026: training on heavily weighted 2023–2024 data means the model learns the *current* competitive landscape rather than historical averages.

### Training Time Breakdown

- **Part A:** Same number of pairs, just different selection — 0% overhead.
- **Part B:** `sample_weight` is just an extra parameter to `fit()` — 0% overhead.
- **Total: ~0%** — these are pure free wins.

## Idea 2: Time-Gap Features

**Category:** Feature Engineering
**Expected AUC gain:** +0.003 to +0.012
**Training time impact:** +10%

### What

The `time_str` field exists in every result row but is currently unused — only rank positions are extracted. Rank loses critical information: finishing 2nd by 0.01s (sprint finish) vs 2nd by 5 minutes (mountain collapse) are completely different outcomes. Time gaps measure *how competitive* a rider actually was.

### New Features (7 diff features)

| Feature | Description |
|---------|-------------|
| `diff_form_90d_avg_pct_gap` | Average percentage time gap to the winner over the last 90 days. Computed as `(rider_time - winner_time) / winner_time * 100`. Normalising by stage duration makes flat and mountain stages comparable. |
| `diff_form_90d_median_pct_gap` | Median version — robust to single blowup stages where a rider cracked. |
| `diff_career_avg_pct_gap` | Career average percentage gap. Captures long-term competitiveness independent of field strength. |
| `diff_form_90d_close_finishes` | Count of results within 1% of the winner's time in last 90 days. Identifies riders who are consistently *in contention* even if absolute rank fluctuates. |
| `diff_mountain_avg_pct_gap` | Average % gap specifically on mountain stages (`profile_score > 100`). Climbing time gaps are the most discriminating metric in stage racing. |
| `diff_flat_avg_pct_gap` | Same for flat stages. Captures sprint/positioning ability. |
| `diff_time_gap_trend` | Slope of % gap over last 10 races (linear regression). A rider whose gaps are shrinking is improving; growing gaps indicate declining form. More sensitive than counting top-10s. |

### Parsing `time_str`

PCS format is typically `+0:05:23` or `,, s.t.` (same time). Parse by splitting on `:`, converting to seconds. `s.t.` → 0 gap. Winner → 0 gap. DNF/OTL → exclude from computation.

### Why This Helps

Current top features are all rank-based (`diff_form_90d_top10`, `diff_career_top10_rate`). These have a hard ceiling because rank is ordinal — 1st and 2nd are "close" but the model can't distinguish a photo-finish from a 10-minute gap. Time-gap features add a **continuous competitiveness signal** that's orthogonal to rank counts.

Particularly for mountain stages, time gaps are the **gold standard metric** that cycling experts use to assess GC form. Adding these features brings the model closer to how human experts evaluate riders.

### Training Time Breakdown

- **Feature precompute:** Parsing time strings adds ~2–3 min to feature computation (string → seconds conversion per result row, then aggregation).
- **Model training:** 7 new columns have minimal impact on XGBoost fit time.
- **Total: +10%**.

## Idea 3: Team Strength & Leader Context

**Category:** Feature Engineering
**Expected AUC gain:** +0.003 to +0.010
**Training time impact:** +10–15%

### What

The scraper already captures `team_name` and `team_url` per result, but currently **zero team features** are used. Team dynamics are a core mechanism in professional cycling — domestiques sacrifice their own position to protect and help their team leaders.

### New Features (6 diff features)

| Feature | Description |
|---------|-------------|
| `diff_team_startlist_strength` | Mean Elo (or mean `avg_season_points_3yr`) of all teammates entered in the same race. Captures "how strong is the rider's support team?" |
| `diff_team_num_starters` | Count of teammates in the startlist. More teammates = more tactical control of the race. |
| `diff_is_team_leader` | Binary: 1 if this rider has the highest `avg_season_points_3yr` among their team's starters in this race, 0 otherwise. Leaders get team support; domestiques sacrifice position. |
| `diff_team_roster_depth` | Number of teammates in the top-50 of the PCS individual ranking. Separates strong teams (UAE, Visma-Lease a Bike) from weaker ones at a structural level. |
| `diff_teammate_spec_match` | Count of teammates whose top specialty matches the race profile (e.g., climber teammates in a mountain stage). Having climbing domestiques matters enormously on mountain stages. |
| `diff_team_recent_form` | Mean `form_90d_avg_rank` of teammates. Captures whether the *team* is collectively in good form, not just the individual. |

### Why This Helps

In races like Grand Tour mountain stages, team control determines who can attack and when. A GC contender with 5 strong domestiques controlling the pace is fundamentally different from one riding alone after their team cracked. This is real-world signal with **zero current representation** in the model.

The `is_team_leader` feature is especially valuable — it partially explains why some strong riders finish mid-pack (they were working as domestiques for someone else). Without this feature, the model sees a strong rider's poor result as noise; with it, the poor result is explained.

### Training Time Breakdown

- **Feature precompute:** Requires a startlist lookup per race (join results table for same `stage_url`), then simple aggregation. ~2 min extra during feature computation.
- **Model training:** 6 new columns have minimal impact on XGBoost fit time.
- **Total: +10–15%**.

## Idea 4: Advanced Training Techniques

**Category:** Training Pipeline
**Expected AUC gain:** +0.003 to +0.012 (combined)
**Training time impact:** +5%

### What

Four improvements to the training pipeline:

#### 4a. Feature Selection via Permutation Importance

```python
from sklearn.inspection import permutation_importance

perm = permutation_importance(
    best_xgb, X_test_scaled, y_test,
    n_repeats=10, scoring="roc_auc", random_state=42,
)
# Keep features with mean importance > threshold
keep_mask = perm.importances_mean > 0.0005
selected_features = X.columns[keep_mask]
```

The ablation already showed physical features add noise. Permutation importance identifies *all* near-zero-contribution features, potentially pruning 270 → ~200 features. Fewer features = less noise = better generalisation.

**Training time:** +5 min one-time analysis, then **−5–15% ongoing** (fewer features to evaluate).

#### 4b. Smarter Missing Value Handling

Replace blanket `fillna(0.0)` with domain-appropriate defaults:

```python
# Current (problematic):
df = df.fillna(0.0)

# Proposed (domain-aware):
default_map = {
    "form_30d_avg_rank":  50.0,   # no recent data = average rider
    "form_60d_avg_rank":  50.0,
    "career_avg_rank":    50.0,
    "terrain_same_profile_avg_rank": 50.0,
    "avg_season_points_3yr": 0.0, # no points is truly zero
    # ... etc
}
for col, default in default_map.items():
    if col in df.columns:
        df[col] = df[col].fillna(default)

# Add binary missing indicators for key feature groups
for col in ["form_30d_races", "career_races", "h2h_total_races"]:
    df[f"{col}_missing"] = df[col].isna().astype(float)
```

A missing `form_30d_avg_rank` currently becomes 0.0, which XGBoost interprets as "this rider's average rank in the last 30 days was position 0" — better than 1st place. The correct default is ~50.0 (no data = average). The binary `_missing` indicators let the model learn that "no recent data" is itself informative.

**Training time:** ~0% (code change only).

#### 4c. Race-Tier Weighted Form Features

Weight form features by race tier (WorldTour races count more than lower-tier events):

```python
tier_weights = {".UWT": 3.0, ".Pro": 2.0, ".1": 1.0, ".2": 0.5}

# Weighted average rank over last 90 days
form_90d_weighted_avg_rank = sum(rank * tier_weight) / sum(tier_weight)

# WT-only metrics
form_90d_wt_top10 = count(top-10 finishes in WT races in last 90 days)
career_wt_top10_rate = top-10 count in WT / total WT starts

# Field-adjusted metric
startlist_pctile = rider's percentile rank within current race's startlist
```

Currently all races are weighted equally — a top-10 at a 2.2 race counts the same as a top-10 at a Tour de France stage. The `startlist_pctile` feature is especially valuable: the same rider is a favourite in a weak field and an underdog in a strong one.

**Training time:** +5–10% (simple conditional aggregation during precompute).
