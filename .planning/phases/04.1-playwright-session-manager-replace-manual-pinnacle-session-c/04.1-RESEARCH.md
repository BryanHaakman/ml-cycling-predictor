## Summary

This phase adds an automated Playwright-based session acquisition layer to `data/odds.py`, eliminating the manual `PINNACLE_SESSION_COOKIE` workflow. The Playwright session manager launches headless Chromium, performs a full login to pinnacle.ca (including the "I'm fit to play" responsible gambling gate), extracts the session token from network response headers or cookies, and caches it with an adaptive TTL. A secondary deliverable consolidates scattered constants into a `config.py` and moves all secrets to a `.env` file.

The integration is a new step inserted into the existing `_get_api_key()` chain in `data/odds.py`. The `PINNACLE_SESSION_COOKIE` env var override is preserved as an escape hatch. Session token persistence uses the same JSON-file pattern already established by `data/.pinnacle_key_cache`. The adaptive TTL starts at infinity (no proactive refresh until the first real expiry is observed), then shrinks to match observed Pinnacle session duration.

**Primary recommendation:** Use `playwright` (Python sync API, version 1.58.0) with `sync_playwright()` context manager. Use `page.on("response", ...)` to passively intercept `X-Session` headers from API calls that fire during login. Fall back to `context.cookies()` if header interception misses the token. Detect CAPTCHA/2FA by waiting for an expected post-login DOM element with a short timeout and catching `TimeoutError`. If detection fires, re-launch with `headless=False`.

### Pattern 1: Playwright Sync Session Acquisition

```python
# Source: playwright.dev/python/docs/library + playwright.dev/python/docs/network
from playwright.sync_api import sync_playwright, TimeoutError as PlaywrightTimeoutError

def acquire_session_token(headless: bool = True) -> str:
    """Login to pinnacle.ca and extract X-Session token."""
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=headless)
        context = browser.new_context()
        page = context.new_page()

        # Intercept API responses to capture X-Session header
        captured_token: dict = {}

        def on_response(response):
            token = response.headers.get("x-session")  # headers are lower-cased
            if token and not captured_token.get("value"):
                captured_token["value"] = token

        page.on("response", on_response)

        # Navigate and log in
        page.goto("https://www.pinnacle.ca/")
        page.get_by_label("Username").fill(os.environ["PINNACLE_USERNAME"])
        page.get_by_label("Password").fill(os.environ["PINNACLE_PASSWORD"])
        page.get_by_role("button", name="Sign in").click()

        # Click responsible gambling gate
        # Selector is Claude's discretion — discovered during implementation
        page.wait_for_selector("[data-testid='responsible-gambling-confirm']", timeout=10_000)
        page.click("[data-testid='responsible-gambling-confirm']")

        # Wait for post-login indicator (proves login succeeded)
        try:
            page.wait_for_selector("[data-testid='user-balance']", timeout=15_000)
        except PlaywrightTimeoutError:
            raise CaptchaOrMfaDetected("Post-login element not found — CAPTCHA or 2FA likely")

        # Fallback: extract from cookies if header intercept missed it
        if not captured_token.get("value"):
            cookies = context.cookies()
            for c in cookies:
                if "session" in c["name"].lower():
                    captured_token["value"] = c["value"]
                    break

        browser.close()

    if not captured_token.get("value"):
        raise PinnacleAuthError("Playwright login succeeded but no session token found")

    return captured_token["value"]
```
[CITED: playwright.dev/python/docs/library, playwright.dev/python/docs/network]

**Critical note on headers:** `response.headers` returns header names lower-cased. Always access as `response.headers.get("x-session")`, not `"X-Session"`. [CITED: playwright.dev/python/docs/network, VERIFIED: search results 2026-04-13]

### Pattern 3: Adaptive TTL State Persistence

```python
import json, time, os

SESSION_STATE_PATH = os.path.join(os.path.dirname(__file__), ".pinnacle_session_state.json")
_TTL_INFINITY = float("inf")

def _load_session_state() -> dict:
    """Returns {token, last_acquired, learned_ttl_seconds} or defaults."""
    defaults = {"token": None, "last_acquired": 0.0, "learned_ttl_seconds": _TTL_INFINITY}
    if not os.path.exists(SESSION_STATE_PATH):
        return defaults
    try:
        with open(SESSION_STATE_PATH, "r", encoding="utf-8") as f:
            return {**defaults, **json.load(f)}
    except (OSError, json.JSONDecodeError):
        return defaults

def _save_session_state(state: dict) -> None:
    state_to_save = {k: v for k, v in state.items() if v != _TTL_INFINITY}
    state_to_save["learned_ttl_seconds"] = state.get("learned_ttl_seconds", _TTL_INFINITY)
    try:
        with open(SESSION_STATE_PATH, "w", encoding="utf-8") as f:
            json.dump(state_to_save, f)
    except OSError as e:
        log.warning("_save_session_state: could not write %s: %s", SESSION_STATE_PATH, e)

def _is_session_fresh(state: dict) -> bool:
    """Returns True if cached session is within learned TTL."""
    ttl = state.get("learned_ttl_seconds", _TTL_INFINITY)
    if ttl == _TTL_INFINITY:
        return bool(state.get("token"))
    age = time.time() - state.get("last_acquired", 0.0)
    return age < ttl and bool(state.get("token"))
```

### Anti-Patterns to Avoid

- **Never call `page.on("request")` for the session token** — request headers are outgoing (sent by browser), not the incoming `X-Session` header from Pinnacle's server response. Must use `page.on("response", ...)` instead.
- **Never `sleep()` after login** — use `page.wait_for_selector()` with a timeout. Sleeping wastes time and is fragile if Pinnacle's response time varies.
- **Never store secrets in `config.py`** — config.py is for non-secret constants only. Credentials go in `.env` only.
- **Never call `browser.launch()` outside a `with sync_playwright()` block** — the context manager ensures the browser process is cleaned up even if an exception occurs during login.
- **Never pass `timeout=0` to `wait_for_selector`** — this means "wait forever" and will hang if the selector is never found. Always set an explicit timeout.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Browser automation / login | Custom HTTP session replaying login flow | `playwright` | Pinnacle's login is JavaScript-heavy SPA — raw HTTP sessions cannot execute JS |
| Secret file loading | Custom `.env` parser | `python-dotenv` | Handles quoting, multiline values, comment stripping, override semantics correctly |
| Post-login element waiting | `time.sleep()` + manual DOM polling | `page.wait_for_selector(timeout=N)` | Built-in, raises `PlaywrightTimeoutError` on miss — clean CAPTCHA detection signal |

**Key insight:** Browser automation is fundamentally different from HTTP scripting. Playwright handles JavaScript execution, cookie jar management, redirect following, and SPA navigation transparently — all of which are required for Pinnacle's login flow.

## Common Pitfalls

### Pitfall 1: Response Header Case Sensitivity
**What goes wrong:** `response.headers.get("X-Session")` returns `None` even when the header exists.
**Why it happens:** Playwright normalizes all response header names to lowercase.
**How to avoid:** Always access as `response.headers.get("x-session")`.
**Warning signs:** Token is always `None` despite successful login; works when printing all headers.

### Pitfall 2: Playwright Browser Binaries Not Installed
**What goes wrong:** `playwright install chromium` was never run; import succeeds but `p.chromium.launch()` raises `Error: Executable doesn't exist at [path]`.
**Why it happens:** `pip install playwright` installs the Python bindings only. Binaries are a separate download.
**How to avoid:** Onboarding docs and requirements should note `playwright install chromium` as a one-time post-install step. Existing machine has binaries at `%USERPROFILE%\AppData\Local\ms-playwright\chromium-1194` — confirmed present.
**Warning signs:** Error message mentions `Executable doesn't exist`.

### Pitfall 3: CAPTCHA Detection False Positive — Wrong Post-Login Selector
**What goes wrong:** Login succeeds but the expected post-login DOM element doesn't match the actual Pinnacle markup, so `wait_for_selector()` times out and headed fallback fires every time.
**Why it happens:** The exact selector for the post-login indicator is not known until implementation — it's Claude's discretion.
**How to avoid:** During implementation, inspect the DOM after a successful headed login to identify a stable, unique post-login selector (e.g., a user balance element or navigation item). Use `page.wait_for_url()` as an alternative if Pinnacle redirects to a known post-login URL.
**Warning signs:** Every run triggers headed fallback; headed browser shows a logged-in state.

### Pitfall 4: Token Captured Before "I'm Fit to Play" Click
**What goes wrong:** Network interception captures an early session token that gets invalidated when the user doesn't complete the responsible gambling gate.
**Why it happens:** Pinnacle may issue a preliminary token on credential submit, then a final token after the RG confirmation.
**How to avoid:** Only capture the token after the post-login indicator confirms the full login flow is complete. If multiple `x-session` headers are seen during the flow, keep the last one (overwrite `captured_token["value"]`).
**Warning signs:** Token extracted but API calls immediately return 401.

### Pitfall 5: Headed Browser on Windows Server / Headless Environment
**What goes wrong:** Headed fallback launches but no display is available, causing the browser to crash silently.
**Why it happens:** This is a local dev tool; headed mode requires a display. On the current Windows 10 machine this is not a problem, but it's worth noting for future VPS deployment.
**How to avoid:** The CAPTCHA fallback should log a clear message so the user knows to be at their machine. This is consistent with D-02 (manual trigger only).
**Warning signs:** Browser process starts and immediately exits; no window appears.

### Pitfall 6: `load_dotenv()` Called After Module-Level `os.environ.get()` Uses
**What goes wrong:** Secrets are `None` because `config.py` or `odds.py` read `os.environ.get()` at import time, before `load_dotenv()` is called in `app.py`.
**Why it happens:** Module-level constant assignment (e.g., `KEY = os.environ.get("PINNACLE_USERNAME")`) executes at import time.
**How to avoid:** Do NOT cache env var values as module-level constants in `data/session_manager.py`. Always call `os.environ.get("PINNACLE_USERNAME")` inside the function body (deferred evaluation). `load_dotenv()` in `app.py` before blueprint registration is sufficient.
**Warning signs:** Credentials are always `None` even when `.env` file is correct.

### Pitfall 7: .env Accidentally Committed to Git
**What goes wrong:** Credentials committed to git history.
**Why it happens:** `.env` not in `.gitignore` at the time of first commit.
**How to avoid:** Add `.env` to `.gitignore` and create `.env.example` with dummy values as the committed template. Verify `.gitignore` update is in the same commit as `.env` creation.
**Warning signs:** `git status` shows `.env` as an untracked file.

## Open Questions

1. **How does Pinnacle deliver the X-Session token?**
   - What we know: Phase 4 confirmed the API requires `X-Session` header for authenticated calls. The current code reads it from `os.environ.get("PINNACLE_SESSION")`.
   - What's unclear: Whether the token appears in (a) a response header from the login API endpoint, (b) a cookie named something session-like, (c) a JavaScript variable set post-login, or (d) localStorage.
   - Recommendation: During implementation, run Playwright with `headless=False` and open browser DevTools to trace network activity during login. `page.on("response", ...)` + `context.cookies()` covers (a) and (b). If those miss it, use `page.evaluate("() => localStorage.getItem('session')")` to check (d).

2. **What is the exact DOM selector for the "I'm fit to play" button?**
   - What we know: It exists on the Pinnacle CA login flow per CONTEXT.md specifics.
   - What's unclear: Exact attribute, class, or role. Pinnacle's SPA may change this between deploys.
   - Recommendation: Discover during implementation using headed mode. Prefer `page.get_by_text("I'm fit to play")` or `page.get_by_role("button", name="I'm fit to play")` — these survive CSS refactors.

3. **What is the reliable post-login indicator for CAPTCHA detection?**
   - What we know: Something must change in the DOM after successful login (balance display, avatar, redirect URL).
   - What's unclear: Whether Pinnacle redirects to a predictable URL after login or mutates the same SPA URL.
   - Recommendation: `page.wait_for_url("**/en/sports/**", timeout=15000)` is a common pattern for SPAs that update the path after login. If URL doesn't change, use a balance or header selector.

4. **JSON infinity sentinel for adaptive TTL**
   - What we know: `float("inf")` is not JSON-serializable in Python's stdlib json.
   - What's unclear: Best sentinel value.
   - Recommendation: Use `9_999_999` seconds (~115 days) as the "infinity" sentinel. Any real Pinnacle session will expire far sooner. Store it as an integer in the JSON file; interpret absence of `learned_ttl_seconds` key as infinity as well.
