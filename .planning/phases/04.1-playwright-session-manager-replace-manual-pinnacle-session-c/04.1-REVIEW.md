## Summary

Reviewed the Pinnacle guest API odds client (`data/odds.py`), its test suite (`tests/test_odds.py`), the Flask web app (`webapp/app.py`), plus supporting files (`.gitignore`, `requirements.txt`). The odds module is well-structured with good defensive coding (auth retry, audit logging, "The Field" filtering). The test suite is thorough with 20+ test cases covering the main code paths.

Key concerns: a SQL injection surface in `webapp/app.py`, a division-by-zero bug in `_american_to_decimal`, missing auth check on the second API call in `fetch_cycling_h2h_markets`, and the `odds_log.jsonl` file not being gitignored despite containing potentially sensitive operational data.

## Critical Issues

### CR-01: SQL injection via f-string table name interpolation

**File:** `webapp/app.py:317`
**Issue:** The `api_stats()` endpoint iterates over a hardcoded list of table names and interpolates them into SQL via f-string: `f"SELECT COUNT(*) as c FROM {table}"`. While the table names are currently hardcoded (not user-supplied), this pattern is unsafe by convention and fragile under future maintenance -- if anyone adds user input to the `table` variable or copies this pattern elsewhere, it becomes a direct SQL injection vector. Parameterized queries cannot bind table names, so the fix is to use an allowlist check.
**Fix:**
```python
ALLOWED_TABLES = {"races", "stages", "results", "riders"}
for table in ALLOWED_TABLES:
    row = conn.execute(f"SELECT COUNT(*) as c FROM {table}").fetchone()
    stats[table] = row["c"]
```
The current code already uses a hardcoded list, so the practical risk today is low. However, adding an explicit constant (`ALLOWED_TABLES`) with a type annotation signals intent and prevents accidental expansion.

## Warnings

### WR-01: Division by zero in _american_to_decimal when american=0

**File:** `data/odds.py:106-108`
**Issue:** If `american` is 0 (which is not a valid American odds value but could appear in malformed API data), the function falls through to the negative branch (`american > 0` is False when `american == 0`) and calls `abs(0)`, resulting in `ZeroDivisionError: division by zero` at `100.0 / abs(american)`. Since the input comes from external API data, defensive handling is warranted.
**Fix:**
```python
def _american_to_decimal(american: int) -> float:
  if american == 0:
    raise ValueError("American odds of 0 are invalid")
  if american > 0:
    return round(american / 100.0 + 1.0, 4)
  return round(100.0 / abs(american) + 1.0, 4)
```

### WR-02: No auth check on second API call (markets endpoint)

**File:** `data/odds.py:383-389`
**Issue:** The `fetch_cycling_h2h_markets()` function performs an auth check (`if matchups_resp.status_code in (401, 403)`) on the first call (matchups), and includes retry logic. However, the second call to `/sports/45/markets/straight` (line 383-389) has no auth or status code check at all. If the markets endpoint returns 401/403 or any HTTP error, `markets_resp.json()` will either raise or return unexpected data. The `_check_auth` call on line 371 runs against `matchups_resp` (already checked), not `markets_resp`.
**Fix:** Add auth and status checks after the markets call:
```python
markets_resp = requests.get(...)
_check_auth(markets_resp)
markets_resp.raise_for_status()
markets_data = markets_resp.json()
```

### WR-03: Stale reference to PINNACLE_SESSION_COOKIE in error response

**File:** `webapp/pinnacle_bp.py:36`
**Issue:** The error response on PinnacleAuthError includes `"env_var": "PINNACLE_SESSION_COOKIE"`, but the guest API pivot removed this env var entirely. The odds module no longer uses `PINNACLE_SESSION_COOKIE`. This misleading field will confuse operators debugging auth failures.
**Fix:**
```python
return jsonify({"error": str(e), "type": "auth_error"}), 401
```
Remove the `env_var` key, or update it to reference `data/.pinnacle_key_cache` instead.

### WR-04: _american_to_decimal assumes integer input but API prices may be floats

**File:** `data/odds.py:89`
**Issue:** The type signature declares `american: int`, but the Pinnacle API `price` field (line 423-425) is extracted from JSON, which deserializes numbers as floats by default. If the API ever returns a float price (e.g., `-154.5`), the function would still work numerically but the type contract would be violated. More importantly, the type hint prevents static analysis from catching callers that pass floats.
**Fix:** Widen the type hint to `american: int | float` or add `american = int(american)` at the top of the function to enforce the contract explicitly.

### IN-02: Duplicate Kelly response-building logic in api_predict and api_predict_batch

**File:** `webapp/app.py:181-210` and `webapp/app.py:276-301`
**Issue:** The Kelly analysis response dict construction is duplicated between the single-predict and batch-predict endpoints (approximately 30 lines each). This violates DRY and means any future change to the Kelly response format must be applied in two places. This mirrors the known issue documented in CLAUDE.md about interaction features being duplicated in 3 places.
**Fix:** Extract a helper function:
```python
def _kelly_to_dict(k, odds: float) -> dict:
    return {
        "edge": round(k.edge, 4),
        "edge_pct": f"{k.edge:.1%}",
        "full_kelly": round(k.kelly_fraction, 4),
        "half_kelly": round(k.half_kelly, 4),
        "quarter_kelly": round(k.quarter_kelly, 4),
        "expected_value": round(k.expected_value, 4),
        "should_bet": k.should_bet,
        "bookmaker_odds": odds,
        "implied_prob": round(decimal_odds_to_implied_prob(odds), 4),
    }
```

### IN-03: Bare `except Exception` in api_predict leaks internal details

**File:** `webapp/app.py:214-216`
**Issue:** The catch-all `except Exception as e` returns `str(e)` to the client. In production, this may leak internal stack traces, file paths, or database details to API consumers. The `log.exception()` call is good for server-side logging, but the client-facing message should be generic.
**Fix:**
```python
except Exception:
    log.exception("Prediction error")
    return jsonify({"error": "Internal prediction error"}), 500
```
