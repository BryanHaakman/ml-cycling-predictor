# Phase 04.1: Playwright Session Manager — Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the manual `PINNACLE_SESSION_COOKIE` env var workflow with an automated Playwright-based session acquisition system. When the user clicks "Load from Pinnacle", the system checks a cached session token against an adaptive TTL, and if stale, launches Playwright to log in headlessly, click the "I'm fit to play" responsible gambling button, extract the session token, cache it, and proceed to fetch odds — all in one button press.

Also includes: centralized config file (`config.py` or similar) consolidating scattered constants, and a `.env` file for all secrets/credentials.

**This phase does NOT:**
- Change the Pinnacle API client logic (fetch cycle, market parsing) — only the auth acquisition layer
- Build any new UI — the existing "Load from Pinnacle" button triggers the session manager transparently
- Run predictions or modify the feature pipeline

</domain>

<decisions>
## Implementation Decisions

### Session Acquisition Flow
- **D-01: Headless-first with headed fallback.** Playwright launches headless Chromium, navigates to pinnacle.ca, logs in with stored credentials, clicks the "I'm fit to play" responsible gambling confirmation button, then extracts the session token from network requests/cookies. If CAPTCHA or 2FA is detected, re-launches the same page in a visible (headed) browser for the user to solve manually, then extracts the token and closes.
- **D-02: Manual trigger only.** Session acquisition only runs when the user clicks "Load from Pinnacle" (or when a 401/403 triggers reactive refresh). No background process, no scheduled refresh. This keeps bot detection risk minimal since it's infrequent, user-initiated.
- **D-03: Anti-detection is low priority.** Since triggers are manual and infrequent, realistic browser fingerprints (Playwright defaults) are sufficient. No need for stealth plugins or proxy rotation.

### Credential Management
- **D-04: Single `.env` file for all secrets.** Create a `.env` file at project root containing `PINNACLE_USERNAME`, `PINNACLE_PASSWORD`, and any other secrets. Add `.env` to `.gitignore`. Use `python-dotenv` (or equivalent) to load into the environment. Existing `os.environ.get(...)` calls continue to work unchanged.
- **D-05: `PINNACLE_SESSION_COOKIE` env var stays as manual override.** If set in `.env` or environment, it takes priority over the automated Playwright flow. This is a debugging/escape hatch only — not the expected path.
- **D-06: Centralized config file.** Create a `config.py` (or `config.yaml` + loader) as the single source of truth for non-secret configuration: API base URLs, timeouts, sport IDs, file paths, thresholds, etc. Migrate Pinnacle-related constants from `data/odds.py` first. Other modules can adopt incrementally.

### Session Lifecycle — Adaptive TTL
- **D-07: Adaptive TTL that learns from real session expiry.** The TTL starts at infinity (no proactive refresh on first use). Each time a session is used:
  - If cached session age < TTL → try it. If it works, no action. If 401/403 → Playwright refreshes, and TTL is updated to the session age at which it failed (the new known minimum expiry).
  - If cached session age > TTL → skip trying, Playwright refreshes proactively.
  - Over time, the TTL converges on the real Pinnacle session duration.
- **D-08: TTL persisted to disk.** The adaptive TTL value is saved to a state file (e.g., `data/.pinnacle_session_state.json` with `last_acquired` timestamp and `learned_ttl_seconds`) so it survives app restarts and keeps learning across sessions.
- **D-09: Check-on-trigger, not background timer.** TTL staleness is checked at the moment the user clicks "Load from Pinnacle". No background threads or schedulers.

### Integration with odds.py
- **D-10: New step in the existing auth lookup chain.** The updated `_get_api_key()` chain becomes:
  1. `PINNACLE_SESSION_COOKIE` env var override (manual escape hatch)
  2. Disk cache + adaptive TTL check → use cached token if fresh
  3. Playwright session acquisition → headless login, extract token, update cache + TTL
  4. JS bundle extraction → last-resort fallback for the API key
  5. All exhausted → raise `PinnacleAuthError`
- **D-11: Session token is machine-managed, not in `.env`.** The `.env` file holds credentials (username/password). The session token lives in the disk cache (`data/.pinnacle_session_cache` or similar), written and refreshed automatically by the Playwright session manager.
- **D-12: On 401/403 during fetch: invalidate cache, update adaptive TTL, re-run from step 3.** Consistent with existing retry behavior but now triggers Playwright instead of just JS bundle re-extraction.

### Claude's Discretion
- Exact Playwright selectors for the login form and "I'm fit to play" button (discovered during implementation)
- How CAPTCHA/2FA is detected (page content heuristics, timeout waiting for expected post-login element, etc.)
- Session token extraction method (intercept network requests for X-Session header, read cookies, or inspect localStorage)
- Whether `config.py` uses a class, module-level dict, or dataclass pattern — whatever fits the existing codebase style

</decisions>
