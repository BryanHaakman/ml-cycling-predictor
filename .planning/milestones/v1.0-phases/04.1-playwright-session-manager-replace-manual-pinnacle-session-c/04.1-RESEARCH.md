# Phase 04.1: Pinnacle Guest API — Research (Pivot: No-Login Odds Scraping)

**Researched:** 2026-04-13
**Domain:** Pinnacle guest API (`guest.api.arcadia.pinnacle.com`), H2H odds ingestion without authentication
**Confidence:** HIGH

---

## Context: Why This Research Exists

Phase 04.1 originally targeted Playwright login automation to extract a session token. That approach was abandoned due to GeoComply, reCAPTCHA, and dual-header auth complexity. Both Plan 01 (config.py, .env infrastructure) and Plan 02 (session_manager.py Playwright implementation) are **already implemented and committed**. This research investigates whether the guest API endpoints observed in browser DevTools return H2H odds **without any login or session token**, making the entire Playwright session layer unnecessary.

---

## Executive Summary

**The guest API approach works.** `guest.api.arcadia.pinnacle.com/0.1/` returns complete H2H cycling matchup data — rider names, American odds, cutoff timestamps — with **no session token and no login required**. The endpoints are publicly cached by CloudFlare (`cache-control: public`) and respond in ~150ms.

The existing `data/odds.py` implementation calls the wrong base URL (`api.arcadia.pinnacle.com`, which requires auth) instead of the guest subdomain (`guest.api.arcadia.pinnacle.com`, which is public). The fix is a single constant change in `config.py` plus removal of all session token machinery from `fetch_cycling_h2h_markets()`.

The `data/session_manager.py` and adaptive TTL infrastructure can be retained as dead code or removed — it is no longer needed for odds fetching.

**Primary recommendation:** Change `PINNACLE_API_BASE` in `config.py` to `https://guest.api.arcadia.pinnacle.com/0.1`, remove the `X-Session` header from `fetch_cycling_h2h_markets()`, and add a "The Field" participant filter to exclude rider-vs-field matchups from H2H output.

## Research Question Answers

### Q1: Page Structure — Initial HTML or JavaScript-Rendered?

[VERIFIED: live endpoint test 2026-04-13]

`https://www.pinnacle.ca/en/cycling/matchups/` is a React SPA. Odds data is NOT in the initial HTML — it is fetched by JavaScript from the API after page load. Cloudscraper would return an empty SPA shell.

**Conclusion:** HTML scraping is not viable. API approach is required.

### Q2: Discovery Flow — How to Find Race Links from Main Page

[VERIFIED: live endpoint test 2026-04-13]

The discovery flow via API is:

```
GET https://guest.api.arcadia.pinnacle.com/0.1/sports/45/leagues?all=false
```

Response (verified live, 2026-04-13):
```json
[
  { "id": 285810, "name": "O Gran Camino", "group": "Spain", "matchupCount": 9 },
  { "id": 285816, "name": "O Gran Camino - Stage 1", "group": "Spain", "matchupCount": 10 }
]
```

Each object is a "league" (Pinnacle's term for a race event). The `id` is the key used to fetch matchups and markets for that race. No HTML scraping of race links is needed.

**Optimization:** The sport-level endpoints (`/sports/45/matchups` and `/sports/45/markets/straight`) return all cycling matchups and markets in a single call each, eliminating the per-league iteration loop entirely. See Q3.

### Q3: Odds Extraction — HTML Structure for H2H Matchup Odds

[VERIFIED: live endpoint test 2026-04-13]

There is no HTML structure — odds come from two JSON API endpoints:

**Option A (current code pattern — per-league):**
```
GET /leagues/{league_id}/matchups          → participant names
GET /leagues/{league_id}/markets/straight  → American odds keyed by matchupId
```

**Option B (optimal — sport-level, 2 calls total):**
```
GET /sports/45/matchups?withSpecials=false          → ALL cycling matchups with names
GET /sports/45/markets/straight?primaryOnly=false&withSpecials=false → ALL cycling odds
```

Option B is strictly better: 2 HTTP calls versus 2N calls (N leagues). Both verified working.

**Verified matchup response structure:**
```json
{
  "id": 1628369752,
  "participants": [
    { "alignment": "home", "name": "Ivan Romeo", "order": 0 },
    { "alignment": "away", "name": "Txomin Juaristi", "order": 1 }
  ],
  "league": { "id": 285810, "name": "O Gran Camino" },
  "status": "pending",
  "type": "matchup"
}
```

**Verified market response structure:**
```json
{
  "matchupId": 1628369752,
  "status": "open",
  "type": "moneyline",
  "prices": [
    { "designation": "home", "price": -469 },
    { "designation": "away", "price": 255 }
  ],
  "cutoffAt": "2026-04-14T11:25:00+00:00"
}
```

`prices[].price` is American odds (integers). This maps directly to `_american_to_decimal()` in `data/odds.py`.

### Q4: JavaScript Rendering — Does CloudFlare/JS Gating Block the API?

[VERIFIED: live endpoint test 2026-04-13 — direct curl with no headers]

The guest API endpoints are:
- Behind CloudFlare CDN (`server: cloudflare`)
- Cached publicly (`cache-control: public, max-age=656, must-revalidate`)
- **No bot detection active on guest endpoints** — responded 200 to curl with no headers at all
- `access-control-allow-origin: *` — CORS open

**Verified: calling with no headers, wrong key, or correct key all return identical 200 responses.** The `X-Api-Key` header has no enforcement on guest endpoints (likely a legacy holdover used by the SPA frontend for observability/routing, not actual auth).

### Q5: Guest API Endpoints — Do They Return Odds Without Session Token?

[VERIFIED: live endpoint test 2026-04-13]

**YES — confirmed.** All three relevant endpoints return full data with no session token and no API key:

| Endpoint | Auth Required | Verified |
|----------|--------------|---------|
| `GET /sports/45/leagues?all=false` | None | YES — 200 with no headers |
| `GET /sports/45/matchups?withSpecials=false` | None | YES — 19 matchups returned |
| `GET /sports/45/markets/straight?primaryOnly=false&withSpecials=false` | None | YES — 19 markets returned |
| `GET /leagues/{id}/matchups` | None | YES — per-league variant also works |
| `GET /leagues/{id}/markets/straight` | None | YES — per-league variant also works |

**CRITICAL FINDING:** The current `config.py` has `PINNACLE_API_BASE = "https://api.arcadia.pinnacle.com/0.1"`. This subdomain returns **HTTP 401** without auth. The guest subdomain `guest.api.arcadia.pinnacle.com` is the public-facing variant. Changing this one constant fixes the entire auth problem.

### Q6: Rate Limiting / Anti-Scraping Protections

[VERIFIED: live endpoint test 2026-04-13]

**CloudFlare caching** is the dominant infrastructure:
- `cf-cache-status: HIT` on all observed requests
- `cache-control: public, max-age=466–860` (7–14 minute cache windows depending on endpoint)
- `age: 7–596` (responses often served from CDN cache, sub-50ms)

**No observed rate limiting** on guest endpoints. The `max-age` values indicate Pinnacle intends these to be polled at most every few minutes.

**Practical rate limit recommendation:** Poll no more frequently than every 60 seconds. The cache TTL (7–14 min) means more frequent polling just gets the same cached response anyway. The existing 0.5s per-request rate limiter in the scraper is sufficient; no special handling needed for the odds fetch.

**No Cloudflare Bot Management observed** on guest endpoints (no JS challenge, no CAPTCHA, no `cf-bot-score` header). The `api.arcadia.pinnacle.com` (authed) subdomain may have stricter protection; this is irrelevant since we use the guest subdomain.

### Q7: OddsMarket Dataclass Compatibility

[VERIFIED: codebase inspection of `data/odds.py`]

The `OddsMarket` dataclass requires:
```python
@dataclass
class OddsMarket:
  rider_a_name: str    # participants[0]["name"] where alignment == "home"
  rider_b_name: str    # participants[1]["name"] where alignment == "away"
  odds_a: float        # _american_to_decimal(prices[designation=="home"]["price"])
  odds_b: float        # _american_to_decimal(prices[designation=="away"]["price"])
  race_name: str       # league["name"] from matchup object
  matchup_id: str      # str(matchup["id"])
```

The guest API response maps directly to all six fields. No schema changes needed. The `race_name` is embedded in each matchup object's `league.name` property, so it's available without a separate leagues call.

## Critical Implementation Details

### The Field Filter

[VERIFIED: live data 2026-04-13]

Out of 19 live cycling matchups, 5 contain `"The Field"` as a participant name. These are rider-vs-field outright bets, not H2H matchups. They must be filtered before constructing `OddsMarket` objects.

**Filter logic:** Skip any matchup where either participant name is `"The Field"`.

```python
# In the matchup loop:
participants = matchup.get("participants", [])
if len(participants) < 2:
    continue
rider_a_name = participants[0]["name"]
rider_b_name = participants[1]["name"]
if rider_a_name == "The Field" or rider_b_name == "The Field":
    continue  # outright, not H2H
```

After filtering, 14 of 19 live matchups are genuine H2H matchups.

### Optimal Implementation: 2-Call Pattern

[VERIFIED: live endpoint test 2026-04-13]

The sport-level endpoints return all data in 2 HTTP calls vs 2N calls with the current per-league pattern (N=2 today but could be more during Grand Tour weeks):

```python
# Call 1: all cycling matchups
matchups_resp = requests.get(
    f"{PINNACLE_API_BASE}/sports/{PINNACLE_CYCLING_SPORT_ID}/matchups",
    params={"withSpecials": "false"},
    headers=headers,
    timeout=REQUEST_TIMEOUT,
)

# Call 2: all cycling markets
markets_resp = requests.get(
    f"{PINNACLE_API_BASE}/sports/{PINNACLE_CYCLING_SPORT_ID}/markets/straight",
    params={"primaryOnly": "false", "withSpecials": "false"},
    headers=headers,
    timeout=REQUEST_TIMEOUT,
)
```

Join on `matchup["id"] == market["matchupId"]` (same integer key as current code).

## Recommended Approach: Guest API (No Login)

**Decision:** Use `guest.api.arcadia.pinnacle.com` directly with `requests`. No Playwright, no session token, no login.

**Why this is the right approach:**
1. [VERIFIED] Works today with zero auth. No session to manage, no TTL to track, no expiry.
2. [VERIFIED] 150ms per call, CloudFlare cached. Faster than any Playwright-based approach.
3. Simplifies `fetch_cycling_h2h_markets()` dramatically — removes ~60 lines of session management.
4. Eliminates the entire `data/session_manager.py` dependency from the hot path.
5. No Pinnacle account required for read-only odds access.

**What changes in `data/odds.py`:**
- `PINNACLE_API_BASE` in `config.py` changes to `guest.api.arcadia.pinnacle.com`
- Remove `X-Session` header from requests in `fetch_cycling_h2h_markets()`
- Remove `get_session_token()` call (or make it optional — see below)
- Replace 3-step per-league loop with 2-call sport-level fetch
- Add "The Field" participant filter

**What stays the same:**
- `OddsMarket` dataclass — unchanged
- `_american_to_decimal()` — unchanged
- `_append_audit_log()` — unchanged
- `_get_api_key()` and JS bundle extraction — can be retained or deprecated; the static key is still included in headers as a courtesy (no-op on guest endpoints) to match browser behavior
- `PinnacleAuthError` — retained for future-proofing; guest endpoints don't trigger it in practice

**Session manager status:** `data/session_manager.py` remains on disk (already committed). It is no longer called in the hot path. The `PINNACLE_SESSION_COOKIE` env var override (D-05) can be preserved as an escape hatch but is effectively unused.

## Pitfalls and Risks

### Pitfall 1: Wrong API Subdomain

**What goes wrong:** Calling `api.arcadia.pinnacle.com` instead of `guest.api.arcadia.pinnacle.com` returns HTTP 401.
**Status:** This is the current bug in `config.py`.
**Fix:** One-line constant change. Test with a bare `requests.get()` to confirm 200.

### Pitfall 2: "The Field" Contaminating H2H Results

**What goes wrong:** `OddsMarket` objects with `rider_b_name="The Field"` appear in results. These are outright winner bets, not H2H matchups. The prediction model has no reference data for "The Field" as a rider.
**Fix:** Explicit name-equality filter before constructing `OddsMarket`.
**Warning signs:** `rider_b_name == "The Field"` in any returned market.

### Pitfall 3: Guest API Subdomain Stability

**What goes wrong:** Pinnacle restructures their API and the guest subdomain changes or adds auth.
**Risk level:** LOW — this subdomain has been observed in open-source projects dating back to at least 2023. CloudFlare caching with `cache-control: public` suggests intentional public exposure.
**Mitigation:** The `_get_api_key()` and JS bundle extraction code provides a fallback if the key becomes required. Log HTTP status on every fetch; alert on non-200.

### Pitfall 4: X-Api-Key Key Rotation

**What goes wrong:** The static key `CmX2KcMrXuFmNg6YFbmTxE0y9CIrOi0R` is rotated and even the guest endpoints start rejecting it.
**Risk level:** LOW — tested with wrong key AND no key, both return 200. The key appears non-enforced on guest endpoints.
**Mitigation:** Retain `_get_api_key()` and `_extract_key_from_bundle()` as fallback infrastructure. The function already handles extraction from the JS bundle if the cached key fails.

### Pitfall 5: Cache Staleness on Rapid Re-fetch

**What goes wrong:** Two fetches within a 7–14 minute window return identical (cached) odds, missing a line move.
**Risk level:** MEDIUM for intraday trading; LOW for pre-race morning fetch.
**Impact:** For PaceIQ's use case (manual pre-race odds check), this is acceptable — line moves within the cache window are minor.
**Mitigation:** No action needed for current use case. If real-time line monitoring is added in future, add `Cache-Control: no-cache` request header.

### Pitfall 6: Session Manager Tests Now Test Unused Code

**What goes wrong:** `tests/test_odds.py` `TestSessionTokenInFetch` and `TestInvalidateSessionOn401` test X-Session integration that no longer runs in the guest API path.
**Fix:** Update these tests to assert X-Session is NOT sent in the new implementation. Or mark them as legacy.
**Risk:** Failing tests on a path that's no longer executed would be misleading. Needs cleanup.

## Key Technical Decisions Required

1. **Keep or remove `data/session_manager.py` from the active call path?**
   - Recommendation: Retain the file (already committed, tests pass), but remove the `get_session_token()` call from `fetch_cycling_h2h_markets()`. Makes the session manager dead code — document this in the module docstring.

2. **Keep or remove `X-Session` header from requests?**
   - Recommendation: Remove it. The guest endpoint ignores it; including it adds unnecessary complexity and keeps session manager tests in an inconsistent state.

3. **2-call sport-level or N-call per-league pattern?**
   - Recommendation: 2-call sport-level. It is strictly better (fewer calls, same data). The per-league pattern is a premature optimization for a use case that doesn't exist (querying individual race odds in isolation).

4. **Keep `X-Api-Key` header in requests?**
   - Recommendation: Yes — keep it as a courtesy header matching browser behavior. It's harmless on guest endpoints and provides a graceful upgrade path if the key becomes required in future.

5. **Update tests for the new sessionless flow?**
   - Recommendation: Yes — `TestSessionTokenInFetch` and `TestInvalidateSessionOn401` in `test_odds.py` test behavior that no longer runs. They should be updated or removed to avoid misleading coverage signals.
