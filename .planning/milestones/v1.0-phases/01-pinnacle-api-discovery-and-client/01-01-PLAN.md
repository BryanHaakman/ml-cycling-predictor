**Section 1: Base URL and Authentication**
- Base URL: `https://guest.api.arcadia.pinnacle.com/0.1`
- Required headers:
  ```
  X-Api-Key: {value — see Key Extraction below}
  Referer: https://www.pinnacle.com/
  Accept: application/json
  ```
- Auth behavior table (from RESEARCH.md §Auth Behavior Summary):
  - No key + public league → 200
  - No key + gated league → 401 `{"status": 401, "detail": "No authorization token provided"}`
  - Invalid key → 403 `{"status": 403, "title": "BAD_APIKEY"}`
  - Valid key → 200, all 65 cycling matchups accessible

**Section 2: API Key Extraction (JS Bundle)**
- The X-Api-Key is a guest token embedded in Pinnacle's frontend JS bundle.
- Runtime extraction approach: fetch `https://www.pinnacle.com/` HTML, locate the
  main JS bundle `<script src="...">` tag (the largest/versioned bundle), fetch that JS
  source, and regex-match for a 32-character alphanumeric string that follows a known
  key-assignment pattern. Record the exact regex pattern found during Playwright
  inspection here once confirmed.
- Cache path: `data/.pinnacle_key_cache` (gitignored plain-text file, one line)
- Lookup order: `PINNACLE_API_KEY` env var → `data/.pinnacle_key_cache` → JS bundle extraction
- The key may rotate when Pinnacle deploys frontend updates.

**Section 3: Cycling Sport ID**
- Sport ID: `45` (confirmed via `/0.1/sports` endpoint)
- Endpoint to list active cycling leagues:
  `GET /0.1/sports/45/leagues?all=false`

**Section 4: Fetch Endpoints**
```
GET /0.1/sports                                    -> list of all sports
GET /0.1/sports/45/leagues?all=false               -> active cycling leagues (today only)
GET /0.1/leagues/{league_id}/matchups              -> rider names and matchupId
GET /0.1/leagues/{league_id}/markets/straight      -> moneyline odds by matchupId
```

**Section 5: Active Cycling Leagues (as of 2026-04-11)**
Include the table from RESEARCH.md:

| 263773 | Paris-Roubaix - Women | 14 |
| 234846 | Itzulia Basque Country - Stage 6 | 13 |

**Section 6: Odds Format**
- Pinnacle returns **American integer odds** (e.g., `-154`, `+107`)
- Conversion to decimal:
  ```
  positive: decimal = american / 100.0 + 1.0     # +107 -> 2.07
  negative: decimal = 100.0 / abs(american) + 1.0 # -154 -> 1.6494
  ```
- `data/odds.py` normalizes to decimal before returning. Callers never see American odds.
- `data/odds_log.jsonl` also records post-normalization decimal odds (never American).

**Section 7: Full Example Responses**
Paste both verbatim JSON examples from RESEARCH.md:
- Full matchup response (id: 1628017725, Tomas Kopecky vs Brent van Moer)
- Full straight market response (prices: home -154, away +107)

**Section 8: Join Key**
- `market["matchupId"] == matchup["id"]` — both integers in the API; cast to `str` when
  building `OddsMarket.matchup_id` (per D-06)
- Only build an `OddsMarket` when both matchup AND market exist and market status is "open"

**Section 9: Delta Updates (future use)**
- `?version={max_version}` query param returns only changed markets
- API returns HTTP 204 (no content) when nothing has changed
- Relevant for ODDS-04 (Phase 4 refresh-odds endpoint) but not Phase 1

**Section 10: Known Pitfalls**
Include all six pitfalls from RESEARCH.md §Common Pitfalls in condensed form:
1. Non-list responses from gated leagues (check isinstance before iterating)
2. Matchup/market ID mismatch (skip if either side is missing)
3. Filtering suspended markets (status != "open")
4. Circular import from models layer (define _american_to_decimal privately in data/odds.py)
5. JSONL file path (use os.path.dirname(__file__))
6. Thread safety note (debug=False, single-threaded per CLAUDE.md)
